# CONSTITUTION.md

**Project Constitution — Supreme Authority Document**
Version: 1.0 | Status: ACTIVE
Last Updated: 2025-01-01 | Owner: Tech Lead

Any Pull Request violating any rule below SHALL be automatically rejected.

Any exception requires an RFC in `/docs/rfcs/` with approval signed by the Tech Lead.

---

# LAYER 1 — HARD RULES

Violation = Immediate merge block, no exceptions.

---

## SEC-01: Password Storage

Passwords SHALL be hashed using **argon2id** (OWASP recommended) or **bcrypt** with cost >= 12.

Plaintext passwords SHALL NOT be stored, logged, or transmitted under any circumstances or environments.

Secrets SHALL NOT appear in source code, configuration files, or logs.

Secrets may only be referenced through environment variables (`$ENV_VARIABLE`) or a vault solution.

---

## SEC-02: Transport Security

Production environments SHALL enforce HTTPS with TLS 1.2 or higher.

Session cookies SHALL only be transmitted over TLS 1.2+ connections in production.

Session cookies SHALL use `HttpOnly`, `Secure`, and an appropriate `SameSite` policy.

CORS SHALL explicitly specify allowed origins.

Wildcard (`*`) origins are PROHIBITED in production.

---

## SEC-03: Mandatory Authentication

THE system SHALL require authentication and authorization for every data-mutating API endpoint:

* POST
* PUT
* PATCH
* DELETE

Authentication SHALL use Session-Based Authentication with Jakarta Servlet `HttpSession`.

JWT Authentication SHALL NOT be used unless a future RFC is approved by the Tech Lead.

Role-Based Access Control (RBAC) is mandatory for:

* Admin
* Building Management
* Resident
* Financial Manager
* Security Guard

Public endpoints SHALL clearly document the reason for public access within the OpenAPI specification.

---

## SEC-04: Session Rules

Session lifetime SHALL be enforced on the server side.

Expired sessions SHALL be rejected.

Session idle timeout SHALL NOT exceed 30 minutes.

Session identifiers SHALL be generated by the servlet container and SHALL NOT be logged or exposed to application code beyond normal session handling.

The system SHALL rotate the session identifier after successful authentication by invalidating any anonymous/pre-login session and creating a new authenticated session.

Logout SHALL invalidate the server-side session.

---

## SEC-05: Input Validation

THE system SHALL validate and sanitize all input originating from:

* JSP/HTML forms
* Query parameters
* Path variables
* File uploads
* JSON payloads

Validation SHALL exist at:

* Frontend (UX validation)
* Backend (security validation)

SQL Injection SHALL be prevented through:

* PreparedStatement
* Parameterized queries
* ORM-safe query patterns

Raw SQL concatenation with user input is PROHIBITED.

---

## SEC-06: Rate Limiting

Authentication endpoints (`/login`, `/login/2fa`) SHALL be limited to:

**10 requests per minute per IP address.**

Requests exceeding the limit SHALL return:

* HTTP 429
* Retry-After header

---

## SEC-07: File Upload Security

THE system SHALL validate:

* File extension
* MIME type
* File size
* Malware scan result

Only the following file types are permitted:

* PDF
* JPG
* PNG

Executable files SHALL be blocked without exception.

---

## DATA-01: Soft Delete

THE system SHALL use soft-delete for all business-critical entities:

* Resident
* Apartment
* Invoice
* Payment
* Complaint
* Vehicle Registration
* Announcement

Hard-delete is only permitted for:

* Temporary files
* Expired sessions
* Logs older than 90 days

---

## DATA-02: Mandatory Audit Logging

THE system SHALL maintain immutable audit logs for:

* Login/Logout
* Payments
* Invoice creation
* Resident modifications
* Role assignments
* Complaint processing

Audit logs SHALL be immutable.

---

# LAYER 2 — ARCHITECTURAL CONSTRAINTS

Violation = Technical debt ticket must be created and resolved before release.

---

## ARCH-01: Service Boundaries

Services SHALL communicate through the Service Layer and DAO Layer according to a layered architecture.

Direct database access from Controller/Servlet is PROHIBITED.

Exception process:

RFC in `/docs/rfcs/` + approval from Tech Lead.

Backend SHALL follow:

* Controller Layer
* Service Layer
* DAO Layer
* Database Layer

Controller/Servlet SHALL only handle:

* Request/Response
* Authentication
* Authorization
* Basic validation

Business logic SHALL reside within the Service Layer.

DAO Layer SHALL only perform data access operations.

Business logic within JSP is PROHIBITED.

JSP SHALL only be used for:

* UI rendering
* Data presentation
* Form submission
* JSTL/EL expressions

---

## ARCH-02: Frontend & Backend Communication

Frontend JSP SHALL communicate with Servlet Controllers.

AJAX requests (if any) SHALL communicate with backend services through REST APIs.

Direct database access from frontend is STRICTLY PROHIBITED.

API contracts SHALL be documented using OpenAPI/Swagger.

---

## ARCH-03: Asynchronous Operations

Operations with an expected processing time greater than 2 seconds SHALL be processed asynchronously through a message queue (Kafka or RabbitMQ).

Synchronous HTTP calls with timeouts exceeding 2 seconds constitute an architectural violation.

---

## ARCH-04: Idempotency

The following endpoints SHALL implement idempotency mechanisms to prevent duplicate operations:

* Payments
* Invoice creation
* Resident registration

---

## ARCH-05: Error Response Contract

Error responses SHALL use the following structure:

```json
{
  "error_code": "RESOURCE_NOT_FOUND",
  "message": "User-friendly description",
  "request_id": "uuid-v4"
}
```

Stack traces SHALL NOT appear in any client-facing response.

Internal error details SHALL NOT be exposed outside server logs.

---

## ARCH-06: Authentication Architecture

Authentication SHALL use:

* Jakarta Servlet `HttpSession`
* Server-side session storage managed by Apache Tomcat
* Secure HTTP-Only session cookies
* `AuthenticationFilter` for authenticated pages
* `RoleFilter` for role-based authorization

Session Expiration:

* Idle timeout: maximum 30 minutes
* Absolute timeout: define per deployment policy if required

Logout:

* The current session SHALL be invalidated.
* Authenticated pages SHALL reject missing or expired sessions and redirect to `/login`.

# LAYER 3 — ENGINEERING STANDARDS

Violation = PR review comments; must be fixed before merge.

---

## ENG-00: Approved Technology Stack

| Layer              | Technology          | Version   |
| ------------------ | ------------------- | --------- |
| Runtime            | Java                | 17 |
| Framework          | Jakarta Servlet     | >= 6.0    |
| Application Server | Apache Tomcat       | >= 10.1   |
| Database           | SQL Server          | 2019      |
| Data Access        | JDBC                | >= 13.2   |
| Cache              | Optional Redis      | >= 7      |
| Queue              | Optional (RabbitMQ) | >= 5.x    |
| Authentication     | Session-Based Auth  | HttpSession |

Changing the technology stack requires an RFC and Tech Lead approval before implementation.

---

## ENG-01: Test Coverage

THE system SHALL enforce a three-tier testing strategy:

1. **Unit Tests (Kiểm thử mức đơn vị)**: 
   - SHALL maintain a minimum line coverage of 80% for all business logic.
   - Critical flows (authentication, payments, data modifications) SHALL maintain 100% unit test coverage.

2. **Integration Tests (Kiểm thử tích hợp)**: 
   - SHALL be mandatory for all API endpoints.
   - Integration tests MUST cover both happy paths and error paths.

3. **End-to-End (E2E) Tests (Kiểm thử End-to-End)**: 
   - Are optional but strongly recommended for the most critical user flows.

Proof-of-concept code SHALL NOT be merged into main.

Coverage thresholds SHALL be enforced within the CI pipeline.

Builds SHALL fail when coverage drops below the required threshold.

---

## ENG-02: Git Workflow

Direct pushes to `main` and `develop` are PROHIBITED.

Branch naming SHALL use one of the following prefixes:

* feat/
* fix/
* chore/
* hotfix/

Commit messages SHALL follow Conventional Commits:

```text
type(scope): description
```

PRs SHALL require at least one approved review before merge.

PRs SHALL pass all CI checks before merge is allowed.

---

## ENG-03: API Documentation

All public API endpoints SHALL have OpenAPI 3.x documentation.

Documentation SHALL be automatically generated from annotations.

Endpoints without documentation SHALL NOT be merged into main.

---

## ENG-04: Dependency Management

All third-party libraries SHALL be pinned to exact versions.

Version ranges such as `^` and `~` are prohibited.

`pom.xml` SHALL be centrally managed within the repository.

Every dependency SHALL declare a specific version.

Major version upgrades require:

* Security review
* Separate PR

Dependencies SHALL be scanned using:

* OWASP Dependency Check
* Snyk
* Dependabot

HIGH and CRITICAL vulnerabilities SHALL block merge.

---

## ENG-05: Logging Standards

Logs SHALL use structured JSON format.

Only the following log levels are permitted:

* ERROR
* WARN
* INFO
* DEBUG

Personally Identifiable Information (PII) SHALL NOT appear in logs.

Each log record SHALL include:

* timestamp
* level
* request_id
* service

# LAYER 4 — AI AGENT POLICY

Applies to all AI agents (Claude, Copilot, Cursor, etc.) operating within this codebase.

---

## AI-01: Allowed Actions (No Approval Required)

* Create, modify, and refactor application code
* Write or update unit tests and integration tests
* Update OpenAPI documentation and code comments
* Suggest dependency updates (suggest only, do not apply automatically)
* Read any file within the repository
* Create new files under:

  * src/
  * test/
  * docs/

---

## AI-02: Prohibited Actions (Require Explicit User Approval)

Database migrations:

* SHALL NOT create or execute migration files without explicit instructions for each migration.

Schema changes:

* SHALL NOT modify schema definitions (e.g., prisma/schema.prisma) without approval.

Secret management:

* SHALL NOT create, modify, or read `.env` files.
* SHALL NOT hardcode credentials.

Production operations:

* SHALL NOT execute commands in production environments.

Dependency modifications:

* SHALL NOT modify dependencies without explicit approval.

File deletion:

* SHALL NOT delete any files except those under:

  * temp/
  * logs/

Git operations:

* SHALL NOT push, merge, or create tags.
* May only provide recommendations.

Infrastructure changes:

* SHALL NOT modify CI/CD configurations, Dockerfiles, or Infrastructure-as-Code without review.

---

## AI-03: Self-Verification Protocol

Before submitting any code, AI SHALL verify all items below.

### SECURITY CHECKLIST

* [ ] No secrets, API keys, passwords, or IP addresses in plaintext
* [ ] No JWT access tokens, refresh tokens, or JWT secrets are introduced
* [ ] Session cookies use HttpOnly, Secure in production, and an appropriate SameSite policy
* [ ] Passwords are hashed using BCrypt
* [ ] All mutating endpoints require authentication
* [ ] RBAC authorization functions correctly
* [ ] Input validation exists before database operations
* [ ] File upload validation exists
* [ ] No raw SQL with unparameterized user input

### ARCHITECTURE CHECKLIST

* [ ] Layered architecture is respected
* [ ] No business logic inside React UI
* [ ] JSP pages communicate with Servlet Controllers
* [ ] Async operations > 2 seconds use queue/thread workers
* [ ] Payment/Invoice APIs implement idempotency
* [ ] Logout invalidates the HttpSession

### DATA INTEGRITY CHECKLIST

* [ ] No hard-delete on business-critical entities
* [ ] Soft-delete pattern is correctly applied

### ENGINEERING CHECKLIST

* [ ] Tests exist for all new business logic
* [ ] Coverage does not fall below 80%
* [ ] OpenAPI annotations added for new endpoints
* [ ] No new dependency added without approval
* [ ] No `console.log` containing PII

If any checklist item fails:

**STOP. Fix the issue before submission. Bypassing the checklist is not permitted.**
