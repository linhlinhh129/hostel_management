1. # **Giới thiệu** {#giới-thiệu}

   Tài liệu này mô tả các yêu cầu phần mềm cho Hệ thống Quản lý Nhà Trọ. Tài liệu được tổ chức theo cấu trúc SRS chuẩn, bao gồm mô tả tổng quan hệ thống, các tính năng chức năng theo từng role người dùng, yêu cầu dữ liệu, giao diện ngoài và thuộc tính chất lượng.

1. ## **Mục đích** {#mục-đích}

   Tài liệu này đặc tả các yêu cầu phần mềm cho **Hệ thống Quản lý Nhà Trọ** (Hostel Management System), phiên bản 1.0. Hệ thống được xây dựng nhằm số hóa và tự động hóa toàn bộ quy trình vận hành chuỗi nhà trọ, bao gồm: quản lý cơ sở và phòng, ký kết hợp đồng, theo dõi người thuê, ghi chỉ số điện/nước, lập hóa đơn, thu tiền trực tuyến qua VNPay, gửi thông báo, xử lý yêu cầu sự cố và báo cáo doanh thu.

   Tài liệu hướng đến các đối tượng đọc sau:
   - **Nhà phát triển phần mềm**: hiểu rõ yêu cầu chức năng và phi chức năng để lập trình.
   - **Kiểm thử viên (QA/Tester)**: xây dựng kịch bản kiểm thử dựa trên use case.
   - **Quản lý dự án**: giám sát phạm vi và tiến độ.
   - **Người dùng cuối** (Admin, Manager, Operator, Tenant): xem xét và phê duyệt yêu cầu.
   - **Giảng viên hướng dẫn**: đánh giá mức độ hoàn chỉnh của sản phẩm.

2. ## **Quy ước tài liệu** {#quy-ước-tài-liệu}

   - **UC-XX-NN**: Mã use case, trong đó XX là mã module (AUTH, FAC, PERS, CON, METER, INV, PAY, NOTIFY, REQ, REV, AUD) và NN là số thứ tự.
   - **[BẮT BUỘC]**: Yêu cầu bắt buộc phải thực hiện trong phiên bản này.
   - **[TÙY CHỌN]**: Yêu cầu có thể triển khai trong phiên bản tiếp theo.
   - Các bảng use case sử dụng định dạng chuẩn gồm: UC ID, Actor, Trigger, Mô tả, Điều kiện tiên quyết, Hậu điều kiện, Luồng chính, Luồng thay thế, Ngoại lệ.
   - **ADMIN / MANAGER / OPERATOR / TENANT**: viết hoa để chỉ vai trò người dùng trong hệ thống.

3. ## **Phạm vi dự án** {#phạm-vi-dự-án}

   Hệ thống Quản lý Nhà Trọ là ứng dụng web được phát triển mới hoàn toàn, phục vụ việc quản lý chuỗi nhà trọ/phòng trọ có nhiều cơ sở tại Việt Nam.

   **Các tính năng chính trong phiên bản 1.0:**
   - Xác thực và phân quyền theo 4 vai trò: ADMIN, MANAGER, OPERATOR, TENANT.
   - Quản lý cơ sở (nhà trọ): tạo, kích hoạt, vô hiệu hóa.
   - Quản lý phòng: tự động sinh phòng khi kích hoạt cơ sở.
   - Quản lý nhân sự (Manager, Operator): tạo tài khoản, gán cơ sở, khóa/mở.
   - Ký kết và quản lý hợp đồng thuê phòng.
   - Quản lý người thuê và người phụ thuộc.
   - Ghi chỉ số điện/nước (kèm ảnh xác thực).
   - Lập hóa đơn hàng kỳ với snapshot giá tại thời điểm lập.
   - Thanh toán trực tuyến qua VNPay (two-step: Tenant tạo → Manager duyệt).
   - Gửi thông báo đến toàn hệ thống / cơ sở / phòng.
   - Tiếp nhận và xử lý yêu cầu sự cố từ người thuê.
   - Báo cáo doanh thu toàn hệ thống, theo cơ sở, theo kỳ.
   - Nhật ký kiểm tra hoạt động (Audit Log).
   - Cấu hình hệ thống (VNPay, Email SMTP).

2. # **Mô tả tổng quan** {#mô-tả-tổng-quan}

1. ## **Góc nhìn sản phẩm** {#góc-nhìn-sản-phẩm}

   Hệ thống Quản lý Nhà Trọ là một sản phẩm phần mềm mới được phát triển từ đầu, thay thế cho quy trình quản lý thủ công (sổ sách, Excel). Hệ thống hoạt động như một nền tảng Property Management System (PMS) đa cơ sở, cho phép một Admin quản lý nhiều cơ sở, mỗi cơ sở có một Manager và một Operator phụ trách.

   **Kiến trúc tổng thể:**
   - Frontend: JSP + JSTL, chạy trên Apache Tomcat 10
   - Backend: Java Servlet (Jakarta EE pattern), Java 17
   - Database: Microsoft SQL Server 2022
   - Thanh toán: Tích hợp VNPay API
   - Email: Jakarta Mail / SMTP
   - Triển khai: Docker + Docker Compose

   **Sơ đồ ngữ cảnh hệ thống:**

   ```
   [Người thuê (TENANT)]  ←→  [Hệ thống Quản lý Nhà Trọ]  ←→  [VNPay]
   [Nhân viên (OPERATOR)] ←→  [Hostel Management System]   ←→  [Email SMTP]
   [Ban quản lý (MANAGER)]←→  [Web Application (Tomcat)]   ←→  [SQL Server DB]
   [Quản trị (ADMIN)]     ←→  [hostel-management.war]
   ```

2. ## **Lớp người dùng và đặc điểm** {#lớp-người-dùng-và-đặc-điểm}

   Hệ thống phục vụ 4 lớp người dùng với đặc điểm và quyền hạn khác nhau:

   | Vai trò | Mô tả | Quyền truy cập |
   | :---- | :---- | :---- |
   | **ADMIN** | Quản trị viên cấp cao nhất, quản lý toàn bộ hệ thống | `/admin/*` và toàn bộ |
   | **MANAGER** | Ban quản lý, phụ trách 1 cơ sở cụ thể | `/manager/*`, `/tenant/*` |
   | **OPERATOR** | Nhân viên vận hành kỹ thuật tại cơ sở | `/operator/*` |
   | **TENANT** | Người thuê phòng, đại diện cho 1 phòng | `/tenant/*` |

   **ADMIN** là lớp người dùng ưu tiên cao nhất. Họ không trực tiếp vận hành hàng ngày mà tập trung vào giám sát, cấu hình và báo cáo. **MANAGER** là người dùng tần suất cao nhất, thực hiện hầu hết nghiệp vụ vận hành. **OPERATOR** chủ yếu dùng thiết bị di động khi đi đo đồng hồ. **TENANT** ưu tiên trải nghiệm mobile-friendly.

3. ## **Môi trường vận hành** {#môi-trường-vận-hành}

   **Phía máy chủ:**
   - Hệ điều hành: Linux (Docker container) hoặc Windows Server
   - Application Server: Apache Tomcat 10, port 8080
   - Database: Microsoft SQL Server 2022, port 1433 (Docker port 1434)
   - JVM: Java 17
   - Bộ nhớ: tối thiểu 768MB RAM cho app, 2GB cho DB (Docker)

   **Phía người dùng (client):**
   - Trình duyệt: Chrome 90+, Firefox 90+, Edge 90+, Safari 14+
   - Kết nối Internet ổn định (HTTPS)
   - Thiết bị: máy tính để bàn, laptop, tablet, điện thoại thông minh

   **Môi trường phát triển:**
   - Build tool: Maven 3
   - IDE: IntelliJ IDEA / Eclipse
   - Phiên bản WAR: `hostel-management.war`

4. ## **Ràng buộc thiết kế và triển khai** {#ràng-buộc-thiết-kế-và-triển-khai}

   - Ngôn ngữ lập trình bắt buộc: **Java 17** với Jakarta Servlet API 6.0.
   - View layer: **JSP + JSTL 3.0** (không dùng framework SPA).
   - Database: **Microsoft SQL Server 2022** (các câu lệnh SQL dùng dialect SQL Server: `GETDATE()`, `OFFSET/FETCH`, filtered index, `IDENTITY`).
   - Cổng thông tin thanh toán: **VNPay** (bắt buộc, cấu hình qua `system_config`).
   - Mật khẩu: hash bằng **jBCrypt** (bcrypt), không lưu plaintext.
   - Mọi bảng dữ liệu chính áp dụng **soft delete** (`deleted_at IS NULL`).
   - CSRF protection bắt buộc cho mọi POST request.
   - Mã hóa giao tiếp: HTTPS được khuyến nghị cho môi trường production.
   - Cấu hình VNPay và Email không được hardcode mà lưu trong bảng `system_config`.

5. ## **Giả định và phụ thuộc** {#giả-định-và-phụ-thuộc}

   **Giả định:**
   - Mỗi cơ sở có **đúng 1 Manager** và **đúng 1 Operator** được gán.
   - Mỗi người thuê chỉ thuê **đúng 1 phòng** tại một thời điểm.
   - Chỉ số điện/nước mới luôn **>= chỉ số kỳ trước** (không lùi đồng hồ).
   - Mã cơ sở theo định dạng vùng: HN (Hà Nội), HL (Hải Long), DN (Đà Nẵng)...
   - Tiền phạt trễ hạn: **1% × tiền phòng × số ngày trễ** (tính realtime).
   - Email SMTP và VNPay được cấu hình trước khi hệ thống đi vào hoạt động.

   **Phụ thuộc bên ngoài:**
   - **VNPay Sandbox/Production**: hệ thống phụ thuộc vào tính khả dụng của VNPay API cho luồng thanh toán.
   - **SMTP Server** (Gmail hoặc SMTP doanh nghiệp): cần thiết để gửi email mật khẩu tạm thời và link đặt lại mật khẩu.
   - **Microsoft SQL Server 2022**: bắt buộc (không tương thích với MySQL/PostgreSQL do dùng syntax đặc thù).
   - **Docker**: môi trường triển khai chính thức dùng Docker Compose.

1. ## **Tính năng 1: Xác thực và Phân quyền** {#tính-năng-xác-thực}

   1. ### **Mô tả**

      Module xác thực quản lý đăng nhập, phân quyền theo vai trò, bảo vệ CSRF và luồng đổi mật khẩu bắt buộc. Hệ thống sử dụng Session-based authentication, với `RoleFilter` kiểm tra quyền trên mọi request và `CsrfFilter` bảo vệ mọi POST request.

   2. ### **Chuỗi kích hoạt/Phản hồi**

      | Sự kiện | Phản hồi của hệ thống |
      | :---- | :---- |
      | Người dùng truy cập URL được bảo vệ mà chưa đăng nhập | Chuyển hướng về trang đăng nhập |
      | Người dùng đăng nhập thành công với `force_change_pass = true` | Chuyển hướng đến trang đổi mật khẩu lần đầu |
      | POST request thiếu hoặc sai CSRF token | Trả về HTTP 403 Forbidden |
      | Role không có quyền truy cập URL | Trả về HTTP 403 Forbidden |
      | Manager/Operator thực hiện write khi cơ sở INACTIVE | Trả về flash error, redirect về trang trước |
      | Người dùng đăng nhập thất bại quá nhiều lần | Khóa tài khoản, ghi log |

   3. ### **Yêu cầu chức năng: Đăng nhập**

      | UC ID và Tên: | UC-AUTH-01: Đăng nhập hệ thống |  |  |

      | Tạo bởi: | Nhóm Phát Triển | Ngày tạo: | 09/07/2026 |

      | Kích hoạt: | Người dùng truy cập trang đăng nhập và nhập thông tin |  |  |
      | Mô tả: | Xác thực danh tính người dùng bằng username (email) và mật khẩu, tạo session và chuyển hướng đến dashboard tương ứng với vai trò |  |  |
      | Điều kiện tiên quyết: | Người dùng có tài khoản hợp lệ trong hệ thống với status = 'ACTIVE' |  |  |
      | Hậu điều kiện: | Session được tạo với thông tin UserSessionDTO; người dùng được chuyển hướng đến dashboard theo role |  |  |

6. ## **Tính năng 6: Quản lý Hóa đơn và Thanh toán VNPay** {#tính-năng-hóa-đơn}

   1. ### **Mô tả**

      Manager lập hóa đơn hàng kỳ dựa trên chỉ số điện/nước đã ghi. Hóa đơn snapshot toàn bộ đơn giá tại thời điểm lập. Người thuê thanh toán qua VNPay theo quy trình hai bước: Tenant tạo payment (PENDING) → Manager duyệt (→ PAID hoặc REJECTED).

   2. ### **Chuỗi kích hoạt/Phản hồi**

      | Sự kiện | Phản hồi của hệ thống |
      | :---- | :---- |
      | Tenant click thanh toán hóa đơn UNPAID/OVERDUE | Tạo URL VNPay với HMAC-SHA512, chuyển hướng đến VNPay |
      | VNPay callback với responseCode=00 | Insert payment PENDING (JDBC transaction), redirect về trang chi tiết hóa đơn |
      | VNPay callback với responseCode≠00 | Hiển thị lỗi "Giao dịch không thành công", không tạo payment |
      | Hash VNPay không khớp | Lỗi "Chữ ký bảo mật không hợp lệ" |
      | Manager duyệt payment PENDING | payment → SUCCESS, invoice → PAID |
      | Manager từ chối payment PENDING | payment → REJECTED |
      | Hóa đơn quá hạn | Tính phạt realtime: 1%/ngày × room_fee × số ngày quá hạn |

   3. ### **Yêu cầu chức năng: Thanh toán hóa đơn qua VNPay**

4. ## **Thu thập, toàn vẹn, lưu trữ và xử lý dữ liệu** {#thu-thập-toàn-vẹn-lưu-trữ-và-xử-lý-dữ-liệu}

   **Thu thập dữ liệu:**
   - Dữ liệu người dùng được nhập thủ công bởi Admin/Manager hoặc tự điền từ hệ thống.
   - Chỉ số điện/nước được nhập bởi Operator với xác thực ảnh công tơ.
   - Giao dịch thanh toán được tự động ghi nhận từ callback VNPay.

   **Toàn vẹn dữ liệu:**
   - Sử dụng Foreign Key constraints để đảm bảo tính tham chiếu.
   - CHECK constraint trên bảng `notifications` đảm bảo target_type nhất quán với facility_id/room_id.
   - Unique index `UX_meter_room_date` tránh ghi trùng chỉ số cùng ngày.
   - Unique index `UX_facilities_manager`, `UX_facilities_operator` đảm bảo 1 cơ sở - 1 Manager - 1 Operator.
   - JDBC transaction (setAutoCommit(false)) cho các thao tác đa bảng.
   - Anti-IDOR: mọi truy vấn theo user_id đều join qua ownership.

   **Lưu trữ dữ liệu:**
   - SQL Server 2022 với volume Docker persistent (`hostel_sqlserver_data`).
   - Soft delete trên tất cả bảng chính (`deleted_at IS NULL`).
   - Filtered index trên `deleted_at IS NULL` để tối ưu query.

   **Xử lý dữ liệu nhạy cảm:**
   - Mật khẩu lưu dạng bcrypt hash (jBCrypt), không bao giờ lưu plaintext.
   - Cấu hình VNPay secretKey và SMTP password lưu trong `system_config` DB (không hardcode).
   - Email SMTP password được đọc từ DB, không log ra console.

2. ## **Giao diện phần mềm** {#giao-diện-phần-mềm}

   **VNPay API:**
   - Giao thức: HTTPS POST/GET redirect flow.
   - Xác thực: HMAC-SHA512 với secretKey từ `system_config`.
   - Tham số chính: `vnp_TmnCode`, `vnp_Amount` (×100 để lấy đơn vị xu), `vnp_TxnRef` (format: INV{id}T{timestamp}), `vnp_ReturnUrl`.
   - Callback URL: `/tenant/payment/return` — verify hash trước khi xử lý.

   **Email SMTP (Jakarta Mail):**
   - Giao thức: SMTP/STARTTLS, port 587 (mặc định).
   - Cấu hình từ `system_config`: `email.host`, `email.port`, `email.username`, `email.password`, `email.from`.
   - Gửi bất đồng bộ (thread riêng) để không chặn HTTP response.
   - Hai loại email: mật khẩu tạm thời và link đặt lại mật khẩu (hết hạn 15 phút).

   **JNDI DataSource (SQL Server):**
   - DataSource name: `jdbc/HostelManagement`.
   - Connection pool qua Tomcat `context.xml`.
   - `SET QUOTED_IDENTIFIER ON` tự động sau mỗi lần lấy connection.

3. ## **Giao diện phần cứng** {#giao-diện-phần-cứng}

   Hệ thống không yêu cầu phần cứng đặc biệt ngoài:
   - Máy chủ web/DB: CPU tối thiểu 2 core, RAM 4GB (Docker: app 768MB + DB 2GB).
   - Thiết bị người dùng: bất kỳ thiết bị có trình duyệt web hiện đại và kết nối Internet.
   - Camera trên điện thoại (cho Operator upload ảnh công tơ điện/nước).

4. ## **Giao diện truyền thông** {#giao-diện-truyền-thông}

   - **HTTP/HTTPS**: giao thức chính giữa trình duyệt và Tomcat server.
   - **JDBC over TCP**: giữa Tomcat app và SQL Server 2022 (port 1433).
   - **SMTP/STARTTLS**: gửi email từ server đến người dùng (port 587).
   - **HTTPS**: kết nối từ server đến VNPay API.
   - **Encoding**: UTF-8 trên toàn bộ hệ thống (EncodingFilter đảm bảo `request.setCharacterEncoding("UTF-8")`).
   - **Session timeout**: cấu hình theo mặc định Tomcat (30 phút).

6. # **Thuộc tính chất lượng** {#thuộc-tính-chất-lượng}

1. ## **Khả năng sử dụng** {#khả-năng-sử-dụng}

   - Người dùng mới (Manager, Operator) có thể thực hiện tác vụ cơ bản sau không quá **30 phút** tự tìm hiểu giao diện.
   - Tất cả các thông báo lỗi phải rõ ràng, cụ thể bằng tiếng Việt.
   - Giao diện OPERATOR phải sử dụng được trên điện thoại thông minh với màn hình 5 inch.
   - Hỗ trợ điều hướng bằng bàn phím (Tab order hợp lý) cho ADMIN/MANAGER.
   - Flash message tự động biến mất sau 5 giây hoặc khi người dùng tắt.
   - Empty state có hướng dẫn rõ ràng (VD: "Chưa có cơ sở nào. Nhấn 'Thêm cơ sở' để bắt đầu.").

2. ## **Hiệu năng** {#hiệu-năng}

   - Thời gian phản hồi trang bình thường: < **2 giây** với dữ liệu trung bình (< 1000 phòng, < 10000 hóa đơn).
   - Dashboard Admin/Manager: < **3 giây** (có nhiều aggregate query).
   - Callback VNPay phải được xử lý trong < **5 giây** (VNPay có timeout).
   - Phân trang 10-20 items/trang để tránh query toàn bộ bảng.
   - Sử dụng filtered index trên `deleted_at IS NULL` để tối ưu query soft delete.
   - Email gửi bất đồng bộ, không ảnh hưởng thời gian phản hồi HTTP.

3. ## **Bảo mật** {#bảo-mật}

   - **Xác thực**: Session-based, session invalidate khi đăng xuất.
   - **Phân quyền**: `RoleFilter` kiểm tra role trên mọi request. 4 cấp phân quyền rõ ràng.
   - **CSRF**: `CsrfFilter` kiểm tra token (form parameter hoặc header `X-CSRF-Token`) trên mọi POST.
   - **Mật khẩu**: bcrypt hash (jBCrypt). Không lưu/log plaintext. Chính sách mật khẩu mạnh (PasswordValidator).
   - **Anti-IDOR**: mọi truy vấn theo resource ID đều join qua ownership (VD: `WHERE r.tenant_id = ?`).
   - **SQL Injection**: sử dụng PreparedStatement với tham số (?) tuyệt đối — không string concatenation.
   - **Soft delete**: dữ liệu không bị xóa vật lý, hỗ trợ điều tra sau sự cố.
   - **Audit log**: ghi lại IP, actor, action, old/new value cho mọi thao tác nghiệp vụ quan trọng.
   - **Cấu hình bí mật**: VNPay secretKey, SMTP password lưu trong DB, không commit vào source code.

4. ## **An toàn** {#an-toàn}

   - Dữ liệu tài chính (hóa đơn, thanh toán) chỉ thay đổi qua luồng nghiệp vụ được kiểm soát.
   - Transaction JDBC đảm bảo atomicity cho các thao tác đa bảng (kết thúc thuê, xác nhận thanh toán).
   - Không có chức năng xóa vật lý dữ liệu nghiệp vụ (chỉ soft delete).
   - Backup DB cần được thực hiện định kỳ theo chính sách vận hành.

5. ## **Các thuộc tính khác** {#các-thuộc-tính-khác}

   **Khả năng mở rộng:**
   - Kiến trúc hỗ trợ thêm cơ sở mới mà không cần thay đổi code.
   - Bảng `system_config` dạng key-value cho phép thêm cấu hình mới linh hoạt.
   - Phân quyền theo role cho phép thêm role mới nếu cần.

   **Độ tin cậy:**
   - Ứng dụng xử lý exception gracefully (try-catch + logger), không để stack trace lộ ra client.
   - Rollback transaction khi gặp lỗi DB.
   - Email gửi thất bại chỉ log warning, không làm fail luồng chính.

   **Khả năng bảo trì:**
   - Tách biệt rõ ràng các tầng: Controller (Servlet) → Service → DAO.
   - Sử dụng SLF4J logger thống nhất cho debugging.
   - Docker Compose hỗ trợ triển khai và khởi động lại dễ dàng.


7. # **Yêu cầu quốc tế hóa và địa phương hóa** {#yêu-cầu-quốc-tế-hóa-và-địa-phương-hóa}

   Hệ thống trong phiên bản 1.0 được thiết kế đặc thù cho thị trường **Việt Nam**:

   - **Ngôn ngữ**: Toàn bộ giao diện, thông báo lỗi và email bằng **tiếng Việt**. Encoding UTF-8 trên toàn hệ thống.
   - **Đơn vị tiền tệ**: **VNĐ (Đồng Việt Nam)**, không có ký hiệu thập phân cho số tiền nguyên (VD: 2,500,000 VNĐ).
   - **Định dạng ngày tháng**: `dd/MM/yyyy` (hiển thị) và `yyyy-MM-dd` (input HTML date).
   - **Định dạng thời gian**: `dd/MM/yyyy HH:mm:ss` cho timestamp.
   - **Định dạng kỳ hóa đơn**: `MM/yyyy` (VD: 07/2026).
   - **Số điện thoại**: Chuẩn Việt Nam 10 số (validate `ValidationUtil.isValidVnPhone`).
   - **CCCD/CMND**: Validate định dạng Việt Nam 9 hoặc 12 chữ số (`ValidationUtil.isValidVnIdentity`).
   - **Múi giờ**: GMT+7 (Việt Nam) — VNPay callback sử dụng `TimeZone.getTimeZone("Etc/GMT+7")`.
   - **Cổng thanh toán**: VNPay (cổng thanh toán phổ biến tại Việt Nam).
   - **Định dạng địa chỉ**: Theo chuẩn địa chỉ Việt Nam (số nhà, đường, phường/xã, quận/huyện, tỉnh/thành phố).

   *Phiên bản tương lai có thể xem xét: hỗ trợ đa ngôn ngữ (tiếng Anh), đa tiền tệ nếu mở rộng sang thị trường khác.*

8. # **Yêu cầu khác** {#yêu-cầu-khác}

   **Yêu cầu cài đặt và cấu hình:**
   - Hệ thống triển khai qua Docker Compose: `docker-compose up` để khởi động DB + App.
   - Script khởi tạo DB: chạy `schema.sql` và `seed.sql` khi setup lần đầu.
   - Cấu hình JNDI DataSource trong `META-INF/context.xml`.
   - Admin cần cấu hình VNPay và Email trong `system_config` qua giao diện web trước khi đi vào sử dụng.

   **Yêu cầu pháp lý và tuân thủ:**
   - Thông tin CCCD/CMND của người thuê phải được bảo vệ theo Luật An ninh mạng Việt Nam.
   - Hợp đồng điện tử lưu trữ tối thiểu 5 năm.
   - Lịch sử giao dịch tài chính phải được lưu trữ đầy đủ, không xóa vật lý.

   **Yêu cầu logging và monitoring:**
   - SLF4J logger ghi INFO cho các thao tác nghiệp vụ quan trọng.
   - WARN cho các sự kiện không nghiêm trọng (VD: gửi email thất bại).
   - ERROR cho các lỗi hệ thống (kết nối DB, exception không mong đợi).
   - Audit log bắt buộc cho: tạo/sửa/xóa nhân sự, khóa/mở tài khoản, kết thúc hợp đồng, duyệt/từ chối thanh toán.

   **Yêu cầu backup:**
   - Backup DB tối thiểu mỗi ngày 1 lần.
   - Lưu trữ backup tối thiểu 30 ngày.

**Phụ lục A: Bảng thuật ngữ** {#phụ-lục-a-bảng-thuật-ngữ}

| Thuật ngữ | Định nghĩa |
| :---- | :---- |
| **ADMIN** | Quản trị viên cấp cao nhất của hệ thống, có toàn quyền truy cập |
| **MANAGER** | Ban quản lý, phụ trách vận hành 1 cơ sở nhà trọ cụ thể |
| **OPERATOR** | Nhân viên vận hành kỹ thuật, chịu trách nhiệm đo đồng hồ và xử lý sự cố |
| **TENANT** | Người thuê phòng, đại diện cho 1 phòng trong hệ thống |
| **Cơ sở** | Một tòa nhà/khu nhà trọ với nhiều phòng (bảng `facilities`) |
| **Soft delete** | Xóa mềm: gán giá trị cho cột `deleted_at` thay vì xóa bản ghi vật lý |
| **Force change password** | Cờ `force_change_pass = 1` bắt buộc người dùng đổi mật khẩu lần đầu đăng nhập |
| **Payment two-step** | Quy trình thanh toán 2 bước: Tenant tạo (PENDING) → Manager duyệt (SUCCESS/REJECTED) |
| **Anti-IDOR** | Kỹ thuật bảo mật ngăn người dùng truy cập tài nguyên của người khác bằng cách join qua ownership |
| **VNPay** | Cổng thanh toán trực tuyến phổ biến tại Việt Nam |
| **CSRF** | Cross-Site Request Forgery — tấn công giả mạo yêu cầu từ site khác |
| **HMAC-SHA512** | Thuật toán ký xác thực HMAC với SHA-512, dùng để xác thực callback VNPay |
| **Meter reading** | Chỉ số đồng hồ điện/nước đo được tại một thời điểm (bảng `meter_readings`) |
| **Snapshot** | Bản sao dữ liệu tại thời điểm lập hóa đơn (giá không thay đổi kể cả khi đơn giá gốc thay đổi) |
| **Audit log** | Nhật ký ghi lại mọi hành động quan trọng trong hệ thống với actor, action, timestamp, IP |
| **BCrypt** | Thuật toán băm mật khẩu an toàn (jBCrypt 0.4), có salt ngẫu nhiên |
| **JNDI** | Java Naming and Directory Interface — cơ chế tra cứu DataSource từ Tomcat |
| **WAR** | Web Application Archive — định dạng đóng gói ứng dụng Java web (hostel-management.war) |
| **DRAFT** | Trạng thái nháp của cơ sở — chưa kích hoạt, chưa có phòng |
| **ACTIVE** | Trạng thái hoạt động bình thường |
| **INACTIVE** | Trạng thái vô hiệu hóa — không thể thực hiện write operations |

**Luồng trạng thái Phòng:**
```
AVAILABLE → [tạo hợp đồng] → OCCUPIED
OCCUPIED  → [kết thúc thuê] → AVAILABLE
AVAILABLE/OCCUPIED → [bảo trì] → MAINTENANCE → AVAILABLE
AVAILABLE → [đặt chỗ] → RESERVED → OCCUPIED/AVAILABLE
ACTIVE_FACILITY → INACTIVE → block all writes
```

**Luồng trạng thái Hóa đơn:**
```
[Lập hóa đơn] → UNPAID
UNPAID → [quá hạn] → OVERDUE (+ tính phạt realtime)
UNPAID/OVERDUE → [Tenant thanh toán VNPay] → [tạo Payment PENDING]
Payment PENDING → [Manager duyệt] → Payment SUCCESS → PAID
Payment PENDING → [Manager từ chối] → Payment REJECTED (invoice vẫn UNPAID/OVERDUE)
```
