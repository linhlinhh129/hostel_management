# Forgot Password Email Not Sent Bugfix Design

## Overview

The forgot password functionality fails to send emails to users requesting password recovery. The system displays "Đã gửi" (Sent successfully) message but no email is actually delivered to the user's inbox. Analysis reveals the root cause is an improperly formatted Gmail App Password in `EmailService.java` containing spaces, which prevents SMTP authentication from succeeding. This fix will remove the spaces from the App Password constant and add comprehensive error logging to catch similar authentication issues in the future.

## Glossary

- **Bug_Condition (C)**: The condition that triggers the bug - when a valid registered email requests password reset and EmailService.sendResetLink() is called but fails to authenticate with Gmail SMTP
- **Property (P)**: The desired behavior - email with reset link must be successfully delivered to the user's inbox via authenticated SMTP connection
- **Preservation**: Existing anti-enumeration behavior (generic success message), rate limiting, validation, token TTL, and session revocation must remain unchanged by the fix
- **EmailService.sendResetLink()**: The static method in `src/main/java/com/quanlyphongtro/util/EmailService.java` that sends password reset emails via Gmail SMTP
- **SENDER_PASSWORD**: The Gmail App Password constant in EmailService used for SMTP authentication - currently contains invalid spaces
- **Transport.send()**: JavaMail API method that transmits the email message through authenticated SMTP connection
- **App Password**: A 16-character alphanumeric code generated by Google for applications to access Gmail via SMTP, must not contain spaces

## Bug Details

### Bug Condition

The bug manifests when a user submits a valid registered email address through the forgot password form. The `ForgotPasswordServlet` correctly validates the input, checks rate limits, generates a reset token, constructs the reset link, and calls `EmailService.sendResetLink()`. However, the email sending fails silently because the SMTP authentication is rejected by Gmail due to the malformed App Password containing spaces.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type ForgotPasswordRequest
         where input = { email: String, userExists: Boolean }
  OUTPUT: boolean
  
  RETURN input.email IS NOT NULL 
         AND input.email IS NOT BLANK
         AND input.userExists = true
         AND EmailService.SENDER_PASSWORD contains spaces
         AND SMTP authentication fails
         AND Transport.send() throws AuthenticationFailedException (caught silently)
END FUNCTION
```

### Examples

- **Example 1**: User enters "tenant@example.com" (exists in database) → System displays "Link khôi phục đã được gửi" → Console shows "Đang gửi email thực tế đến: tenant@example.com ..." → SMTP authentication fails → Exception is caught and printed to console → No email delivered
- **Example 2**: User enters "manager@example.com" (exists in database) → Same flow → Console shows authentication error but servlet still forwards to success page → User never receives email
- **Example 3**: User enters "operator@hostel.vn" (exists in database) → Checks inbox, spam, promotions folders → No email found despite success message
- **Edge case**: User enters "nonexistent@example.com" (not in database) → System still displays "Đã gửi" message (expected anti-enumeration behavior) → No email attempt is made (correct preservation)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Anti-enumeration protection: For non-existent email addresses, the system must continue to display the generic "Link khôi phục đã được gửi" message without revealing whether the email exists in the database
- Rate limiting: Users exceeding 3 requests per hour must continue to receive "Bạn đã vượt quá số lần yêu cầu" error via RateLimitManager
- Input validation: Empty or null email inputs must continue to display "Vui lòng nhập địa chỉ email hợp lệ" validation error
- Token lifecycle: Reset tokens generated by ResetTokenManager must continue to have exactly 15 minutes TTL and be stored in memory cache
- Session revocation: After successful password reset (when user clicks the link and sets new password), SessionRegistry must continue to invalidate all existing sessions for that account
- Reset link format: The link construction logic (`req.getScheme() + "://" + req.getServerName() + ...`) must remain unchanged

**Scope:**
All inputs that do NOT involve valid registered email addresses requesting password reset (the bug condition) should be completely unaffected by this fix. This includes:
- Non-existent email addresses (anti-enumeration flow)
- Empty/null email inputs (validation error flow)
- Rate-limited requests (rate limit error flow)
- The token generation and validation logic
- The password reset form display (GET request to /forgot-password)
- The actual password change flow after clicking the reset link

## Hypothesized Root Cause

Based on code analysis, the root cause is **definitely confirmed**:

1. **Malformed App Password (CONFIRMED)**: The `SENDER_PASSWORD` constant in `EmailService.java` is defined as `"ywgq bjng ymfo bpol"` which contains **three spaces**. Gmail App Passwords are 16-character alphanumeric codes that must be written without spaces. When JavaMail's `PasswordAuthentication` object passes this value to Gmail's SMTP server, the authentication fails with `535-5.7.8 Username and Password not accepted` error.

   **Evidence**: Line 17 in EmailService.java:
   ```java
   private static final String SENDER_PASSWORD = "ywgq bjng ymfo bpol";
   ```
   
   **Expected format**: `"ywgqbjngymfobpol"` (16 consecutive characters)

2. **Silent Exception Handling**: The try-catch block in `sendResetLink()` catches all exceptions and only prints them to console (`e.printStackTrace()`), but does not propagate the error back to the servlet. This means the servlet always proceeds to display the success message even when email sending fails. While this is intentional for anti-enumeration, it also masks legitimate SMTP errors during development and debugging.

3. **Misleading Console Output**: The message "Gửi email thành công!" is printed immediately after `Transport.send()` without verifying the send actually succeeded. If an exception occurs inside `Transport.send()`, it's caught and the success message is never reached, but the servlet still shows success to the user.

4. **No Error Logging**: There is no structured error logging (e.g., to `error.log` file) to track email sending failures in production environments where console output may not be accessible.

## Correctness Properties

Property 1: Bug Condition - Email Delivery via Authenticated SMTP

_For any_ input where a valid registered email address requests password reset (isBugCondition returns true with the App Password fix applied), the fixed EmailService.sendResetLink() function SHALL successfully authenticate with Gmail SMTP using the properly formatted App Password, execute Transport.send() without exceptions, and deliver the email containing the reset link to the recipient's inbox within normal email delivery timeframes (typically under 1 minute).

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation - Non-Buggy Input Behavior

_For any_ input where the bug condition does NOT hold (non-existent emails, empty emails, rate-limited requests, or any request that doesn't trigger EmailService.sendResetLink()), the fixed code SHALL produce exactly the same behavior as the original code, preserving anti-enumeration protection, rate limiting enforcement, input validation errors, token TTL of 15 minutes, and session revocation after password reset.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5, 3.6**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct (App Password formatting):

**File**: `src/main/java/com/quanlyphongtro/util/EmailService.java`

**Function**: `sendResetLink(String toEmail, String resetLink)`

**Specific Changes**:

1. **Fix App Password Format (PRIMARY FIX)**: Remove all spaces from the `SENDER_PASSWORD` constant
   - **Current**: `private static final String SENDER_PASSWORD = "ywgq bjng ymfo bpol";`
   - **Fixed**: `private static final String SENDER_PASSWORD = "ywgqbjngymfobpol";`
   - **Rationale**: Gmail App Passwords must be exactly 16 consecutive alphanumeric characters without spaces. The spaces prevent SMTP authentication from succeeding.

2. **Add Detailed Error Logging**: Enhance the catch block to log specific exception types
   - Add logging for `AuthenticationFailedException` to identify auth failures
   - Add logging for `MessagingException` to identify SMTP connection issues
   - Add logging for `UnsupportedEncodingException` to identify encoding issues
   - Log full stack trace to both console and error file for production debugging

3. **Add Pre-Send Validation**: Add debug logging before `Transport.send()` to confirm SMTP session setup
   - Log SMTP host and port being used
   - Log sender and recipient addresses
   - This helps verify configuration before attempting to send

4. **Add Post-Send Confirmation**: Add explicit success logging after `Transport.send()` completes without exception
   - Move "Gửi email thành công!" inside the try block after Transport.send()
   - This ensures success message only appears when send actually completes

5. **Add Configuration Validation (Optional Enhancement)**: Add static initializer to validate App Password format at class loading time
   - Check that SENDER_PASSWORD is exactly 16 characters
   - Check that SENDER_PASSWORD contains no spaces
   - Throw IllegalStateException if validation fails
   - This catches configuration errors immediately at application startup

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code (exploratory bug condition checking), then verify the fix works correctly (fix checking) and preserves existing behavior (preservation checking).

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm the root cause (App Password with spaces causes SMTP authentication failure). If we refute this hypothesis, we will need to re-investigate.

**Test Plan**: Write integration tests that call `ForgotPasswordServlet.doPost()` with valid registered email addresses, capture console output, and inspect for authentication errors. Run these tests on the UNFIXED code (with spaces in App Password) to observe authentication failures and confirm no emails are delivered.

**Test Cases**:

1. **Valid Email - Console Output Test**: Submit valid email "tenant@example.com" → Observe console for "AuthenticationFailedException: 535-5.7.8 Username and Password not accepted" (will fail on unfixed code)

2. **Valid Email - No Delivery Test**: Submit valid email "manager@example.com" → Wait 60 seconds → Check Gmail inbox/spam programmatically via IMAP → Assert no email received (will fail on unfixed code)

3. **Multiple Valid Emails Test**: Submit 3 different valid emails within rate limit → Observe all 3 fail authentication → Confirm pattern is consistent (will fail on unfixed code)

4. **App Password Validation Test**: Write unit test to check `SENDER_PASSWORD.contains(" ")` → Assert true on unfixed code → This directly validates the root cause hypothesis

**Expected Counterexamples**:
- Console output will show "Đang gửi email thực tế đến: [email]" followed by stack trace containing `AuthenticationFailedException`
- Error message will contain "535-5.7.8 Username and Password not accepted" from Gmail SMTP
- No emails will appear in recipient inboxes
- Possible secondary causes if authentication succeeds: network timeout, firewall blocking port 587, but authentication failure is the primary expected issue

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds (valid registered email requests password reset), the fixed function produces the expected behavior (email is successfully delivered).

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := forgotPasswordFlow_fixed(input)
  ASSERT email_delivered_to_inbox(input.email) = true
  ASSERT email_subject_is("Khôi phục mật khẩu - Quản lý Nhà trọ") = true
  ASSERT email_body_contains_reset_link(input.email) = true
  ASSERT console_shows_success_message(input.email) = true
  ASSERT no_authentication_exception_thrown() = true
END FOR
```

**Test Implementation**:

1. **Manual Verification Test**:
   - Apply fix (remove spaces from App Password)
   - Start application server
   - Navigate to forgot password form
   - Enter valid registered email (e.g., "atu023@gmail.com" - a test account)
   - Submit form
   - Check Gmail inbox within 60 seconds
   - Verify email received with correct subject and reset link
   - Click reset link and verify it navigates to password reset form

2. **Automated Integration Test**:
   - Mock Gmail IMAP connection to check inbox
   - Submit POST request to `/forgot-password` with valid email
   - Poll inbox for 60 seconds checking for email arrival
   - Assert email subject matches "Khôi phục mật khẩu - Quản lý Nhà trọ"
   - Assert email body contains reset link matching pattern `/reset-password?token=`
   - Assert console output contains "✓ Gửi email thành công"

3. **SMTP Connection Test**:
   - Write unit test that creates JavaMail Session with fixed App Password
   - Attempt to authenticate with Gmail SMTP
   - Assert authentication succeeds without exceptions
   - This isolates SMTP authentication from full email sending

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold (non-existent emails, empty emails, rate-limited requests), the fixed function produces the same result as the original function.

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT forgotPasswordFlow_original(input) = forgotPasswordFlow_fixed(input)
END FOR

// Specifically:
// - Non-existent email: "Đã gửi" message shown (anti-enumeration)
// - Empty email: "Vui lòng nhập địa chỉ email hợp lệ" error shown
// - Rate-limited: "Bạn đã vượt quá số lần yêu cầu" error shown
// - Token TTL: still 15 minutes
// - Session revocation: still works after password reset
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many test cases automatically across the input domain (valid/invalid emails, edge cases)
- It catches edge cases that manual unit tests might miss (e.g., emails with special characters, very long emails)
- It provides strong guarantees that behavior is unchanged for all non-buggy inputs

**Test Plan**: Observe behavior on UNFIXED code first for non-buggy inputs (anti-enumeration, validation, rate limiting), document the exact responses, then write property-based tests capturing that behavior and verify it remains identical on FIXED code.

**Test Cases**:

1. **Anti-Enumeration Preservation**: 
   - UNFIXED: Submit "nonexistent@example.com" → Observe "Link khôi phục đã được gửi" message displayed
   - FIXED: Submit "nonexistent@example.com" → Assert identical message displayed
   - Property test: Generate 100 random non-existent emails → Assert all show same generic success message

2. **Validation Preservation**:
   - UNFIXED: Submit empty string → Observe "Vui lòng nhập địa chỉ email hợp lệ" error
   - FIXED: Submit empty string → Assert identical error message
   - UNFIXED: Submit null → Observe same validation error
   - FIXED: Submit null → Assert identical error
   - Property test: Generate various invalid inputs (null, empty, whitespace-only) → Assert all show validation error

3. **Rate Limiting Preservation**:
   - UNFIXED: Submit same email 4 times rapidly → Observe "Bạn đã vượt quá số lần yêu cầu" on 4th attempt
   - FIXED: Submit same email 4 times rapidly → Assert identical rate limit error on 4th attempt
   - Property test: Generate random valid emails and submit each 4 times → Assert rate limit enforced consistently

4. **Token TTL Preservation**:
   - UNFIXED: Generate token → Wait 16 minutes → Attempt to use token → Observe "Token hết hạn" error
   - FIXED: Generate token → Wait 16 minutes → Attempt to use token → Assert identical expiration error
   - Unit test: Verify `ResetTokenManager.TOKEN_TTL_MS` remains exactly `15 * 60 * 1000`

5. **Session Revocation Preservation**:
   - UNFIXED: Login on two browsers → Reset password → Verify both sessions invalidated
   - FIXED: Login on two browsers → Reset password → Assert both sessions still invalidated identically
   - Integration test: Verify `SessionRegistry` is still called after password reset

### Unit Tests

- **SMTP Configuration Test**: Verify App Password is exactly 16 characters without spaces after fix
- **Authentication Test**: Test JavaMail Session authentication with fixed App Password succeeds
- **Email Content Test**: Verify email subject, sender, HTML content are correctly formatted
- **Edge Case - Invalid Recipient**: Test sending to malformed email address (should fail gracefully)
- **Edge Case - Null Parameters**: Test sendResetLink(null, "link") and sendResetLink("email", null) handle gracefully

### Property-Based Tests

- **Email Format Validation**: Generate random valid email addresses → Submit each → Assert either email sent (if exists in DB) or generic success message (if not exists)
- **Rate Limit Enforcement**: Generate random email addresses → Submit each 10 times → Assert rate limit enforced at 4th request every time
- **Token Lifecycle**: Generate random user IDs → Create tokens → Verify all expire after exactly 15 minutes
- **Input Sanitization**: Generate random strings (including SQL injection attempts, XSS payloads) as email input → Assert all handled safely

### Integration Tests

- **Full Password Reset Flow**: Register test user → Submit forgot password → Receive email → Click link → Reset password → Login with new password → Verify old sessions invalidated
- **Concurrent Requests**: Submit 10 forgot password requests simultaneously for different users → Verify all valid users receive emails, all invalid users see generic message
- **Network Resilience**: Temporarily disable SMTP connection (firewall rule) → Submit request → Verify error logged to console but user still sees generic success (anti-enumeration maintained)
- **Gmail Rate Limit**: Send 50 emails rapidly → Verify Gmail may delay delivery but no authentication errors occur (if we hit Gmail's sending limit, that's expected behavior)
